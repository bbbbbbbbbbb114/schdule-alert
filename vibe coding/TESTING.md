# 测试与回归测试说明

## 1. 测试范围
- 日程 CRUD：新增、查询、编辑、删除、完成状态切换。
- 提醒功能：
  - 登录后过期提醒。
  - 页面打开时实时提醒。
  - 提醒配置：提前 5 分钟/10 分钟/1 天。
  - 提醒时机：开始/结束/都提醒。
  - 0级弹窗提醒：必须点击确认才关闭，声音单次播放。
- 重复规则：不重复/每天/每周固定日。
- 地点标签：选填，列表与日历详情可见。
- 可视化日历：
  - 时间点/时间段按类型展示。
  - 周期事件实例展开显示。
  - 当前时间红线显示。
  - 不同日程颜色高亮。
  - 点击日历事件展示详情。

## 2. 应用的测试过程 (Testing Process)

测试过程采用了 **自动化核心逻辑测试 + 人工交互验证 + 安全/边界校验** 三重防护。

### 2.1 自动化测试 (Automated Testing)
执行命令：

```powershell
pytest -q
```

当前自动化测试覆盖：
- 应用创建与 SQLite 配置验证（本地数据库验证）。
- 登录认证、日程 CRUD 及提醒接口的基础校验。
- 时间段日程的创建与多端点查询。
- 复杂时间逻辑测算验证：包括提醒配置偏移量（5m/10m/1d）与周期性重复规则（daily/weekly）展开后端引擎计算。

### 2.2 手动流程与 UI/UX 测试 (Manual UI/UX Testing)
1. **视图渲染测试**：登录后新增一条“时间点”或“时间段”日程（每天/每周重复），检查对应数据能否正确在左侧列表与右侧日历中同步展开为对应的多日实例。
2. **交互阻断测试**：新建即将到期的日程，等待前台30秒轮询触发，验证“0级强制弹窗”是否弹出，且在点击“我已确认”前不消失，声音仅单次播放。
3. **操作反馈测试**：测试完成、编辑（标题等）、删除日程时，确认 DOM 节点直接产生即时交互反馈，列表与日历标题和颜色状态能实时同步。
4. **标签与展示测试**：填写地点标签，检查详情弹窗是否正常渲染地点；同时检查日历上的红色当前时间线是否正确显示且随时间滚动。

### 2.3 边界与安全性测试 (Security & Boundary Testing)
1. **越界文本输入**：故意输入超出限制范围的文本（例如标题>120字、正文>10000字，异常长短的密码等），验证应用返回 400 Bad Request 并被正确拦截。
2. **XSS 注入防御**：输入包含恶意 JS 脚本的 HTML 片段（如 `<script>alert(1)</script>`），验证前端是否能依赖 `escapeHtml` 机制自动进行安全转义显示为纯文本。
3. **隔离权限测试**：跨用户/非法时区的 API 并发调用测试，验证后台的 ID 与 `user` 进行双绑后拦截越权访问。

## 3. 应用的回归测试过程 (Regression Testing Process)

在面临“需求变更”或者完成“深度 BUG 修复”后，回归测试用于验证核心体验不被破坏，且回归测试应被强制前置定义。

### 3.1 自动化回归防线
**核心原则**：任何代码变更合并前，首先在终端重新运行 `pytest -q`。
若结果全绿通过，说明底层数据模型、API契约和时间计算逻辑未受致命破坏。

### 3.2 核心手工回归清单 (Regression Checklist)
手工修改相关代码后，需强制进行以下 5 个关键指标项的手动自测复归：
- **R1 (鉴权回归)**：检查登录态持久性，确认登入/登出逻辑无损，未发生越权或会话穿越漏洞。
- **R2 (时区与渲染的精确一致性)**：时间录入与展示无偏移（例如设定 18:22 仍必须显示 18:22），确保前后端的时间戳数据解析无损穿透。
- **R3 (降噪弹窗回归，极度重要)**：定时器防重复轰炸机制。测试前台点击“我已确认”后，同一复合 `reminder_id` 被缓存，再次轮询或刷新页面时绝对不能再次弹出该历史提醒。
- **R4 (业务模块隔离渲染)**：验证周期性日程渲染规则加上新需求后，是否会对单点日程产生污染（例如防范将单点日程连画多天）。
- **R5 (多端/UI响应式排版回归)**：调整不同的浏览器视口宽度，确认 CSS Flexbox/Grid 结构下侧边栏、卡片容器不发生严重的断行与层叠错乱。

### 3.3 遵循 CLAUDE.md 的防退化机制
结合 `CLAUDE.md`，使用“成功导向验证”：
- **前置定义验证目标**：每次需求变更或修复前必须先设立并拟定成功条件。例如若进行最小化切除（Surgical changes），需提前构思：“我要加入用户长度验证 → 试图用超长用户名穿透 → 应当收到前端拦截与后端400”。
- **目标达成闭环**：仅当这些成功标准通过后，本次改动才被记录为彻底结束。这从开发流程基础上规避了不可预期的退化缺陷。
